ℹ️ Note

Le symbole 🚹 signifie qu'une action n'est possible que sur son propre compte.

Par exemple un medecin ayant ✅ 🚹 à l'action Modifier, ne peux que modifier que des informations de son propre compte.

Medecins

Utilisé pour permettre de modifier le contenu en rapport avec les médecins. Chaque requête possède une vérification des roles

Méthodes autorisées :     [GET, POST, PATCH, DELETE, OPTIONS]

GET  /medecins/
Obtenir tout les médecins

- Authentification

Nécessite le role de :  administrateur ,  secrétaire ,  mecedin ,  usager

- Requête

Méthode :    GET

URL :    /medecins/

Header :

Authorization : Bearer eyJhbGc...
Réponse - 200

Header :

Content-Type : application/json

Body :

{
    "status": "success",
    "status_code": 200,
    "status_message": "[R200 REST API] : Médecins trouvés",
    "data": [
        {
            "id_medecin": 1,
            "civilite": "M.",
            "nom": "Dupont",
            "prenom": "Xavier"
        },
        {
            "id_medecin": 2,
            "civilite": "Mme.",
            "nom": "Darc",
            "prenom": "Jeanne"
        },
        ...
    ]
}

Réponse - 404

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 404,
    "status_message": "[R404 REST API] : Aucun médecin trouvé"

POST  /medecins/
Créer un médecin

Envoie une requête au serveur d'authentification pour ajouter le médecin au répertoire de connexion.

- Vérifications

  • Données necessaires : [nom, prenom, civilité, mot de passe]
  • Civilitée

- Authentification

Nécessite le role de :  secrétaire , administrateur

- Requête

Méthode :    POST

URL :    /medecins/

Header :

Content-Type : application/json
Authorization : Bearer eyJhbGc...

Body :

{
    "civilite":"M.",
    "nom":"Dupond",
    "prenom":"Gérard",
    "mdp": "mdp"
}
Réponse - 201

Header :

Content-Type : application/json

Body :

{
    "status": "success",
    "status_code": 201,
    "status_message": "[R201 REST API] : Médecin inséré en base de donnée avec succès",
    "data": {
        "id_medecin": 11,
        "civilite": "M.",
        "nom": "Dupond",
        "prenom": "Gérard",
        "login": "MMODUGE95"
    }
}

Réponse - 500

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 500,
    "status_message": "[R500 REST API] : Erreur lors de l'insertion du médecin en base de donnée"
}

Vérifications

Répertoire de toute les vérifications pouvant être effectué pour chaque requete.

Si une vérification passe avec succès rien n'est renvoyé. On reçoit un message d'erreur dans le cas contraire.


Méthode


Chaque route détient une liste de méthodes autorisée. Cette vérification est la première effectuée à chaque requête.

- Response - 405

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 405,
    "status_message": "[R405 REST AUTH] : Methodes utilisées non autorisées"
}

Argument


Utilisé quand la requete attend un argument que ca soit un {id} ou un nom de fonction tel que {usager} (pour les statistiques).

- Response - 400

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 400,
    "status_message": "[R400 REST API] : Aucun argument n'a été fourni"
}

Roles


Sachant que toutes les requêtes necessite un token, on vérifie à chacune d'entre elles les roles autorisés à effectuer la requête.

- Response - 403

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 403,
    "status_message": "[R403 REST AUTH] : Rôle non autorisé à effectuer cette action."
}

Jeton


Verifie si le jeton est présent dans le header d'autorisation et si ce dernier est valide. La vérification de la validité est effectuée coté serveur d'authentification et l'erreur de validité est retournée coté serveur application.

- Response - 401

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 401,
    "status_message": "[401 REST AUTH] : Jeton requis."
}

- Response - 401

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 401,
    "status_message": "[401 REST AUTH] : Jeton invalide."
}

Données Nécessaires


A partir des données passées en corps, vérifie si touts les champs nécessaires sont présents (surtout pour les requêtes de création).

- Response - 400

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 400,
    "status_message": "[R400 REST API] : Le champ <champs> est requis"
}

Données Autorisées


A partir des données passées en corps, vérifie si touts les champs sont autorisés.

- Response - 400

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 400,
    "status_message": "[R400 REST API] : Le champ <champs> n'est pas autorisé"
}

Element Existe


Utilisé pour vérifier à partir d'un ID si un element existe (en fonction de si la requête est faites pour un médecin, un usager ou une consultations).

- Response - 404

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 404,
    "status_message": "[R404 REST API] : Aucun <element> avec l'ID <id> n'a été trouvé"
}

Contrôle d'Accès


Certaines données ne peuvent être visualisée, modifiée ou supprimée uniquement par un administrateur, une secretaire ou leur propriétaire. On vérifie donc en fonction du role actuel de l'utilisateur si il à le droit d'efffectuer l'action demandée.

De ce fait, un médecin ne pourras par exemple que modifier ses rendez-vous ou alors un usager ne pourras pas visualiser les informations des autres usagers. Voir les tables d'accès

- Response - 403

Header :

Content-Type : application/json

Body :

{
    "status": "error",
    "status_code": 403,
    "status_message": "[R403 REST API] : Vous n'avez pas le droit de visualiser/modifier/supprimer cet <élement>"
}
